Data model

  • School — profile + UPI VPA (fees pay into this directly).
  • User — role: principal | teacher | desk; teachers carry classRoomId (the class they own).
  • ClassRoom → Student — students hold guardianName/guardianPhone (the WhatsApp channel) and language (en|hi, message language per family).
  • AttendanceRecord — one row per student per UTC day, unique(studentId, date); status present | absent | leave.
  • FeeHead → FeeInvoice → Payment — invoice status unpaid | partial | paid derived from payment totals; amounts in whole rupees.
  • Notice — audience "all" or "class:<id>"; class diary entries are notices titled "Class diary".
  • OutboxMessage — every parent-facing message: channel, template, rendered body, status queued | sent | failed. All communication flows through this one table.

Messaging architecture

lib/messaging.ts exposes queueMessage(s) and bilingual templates (fee_reminder, absence_alert, receipt, notice). Nothing sends directly — actions queue, a provider drains. The dev provider marks rows sent so every flow is demoable with zero credentials. Swapping in a real WhatsApp Business API provider (Interakt/Gupshup/Twilio or Meta Cloud API direct) means implementing one drain function plus webhook status updates; templates map 1:1 to WABA message templates that need Meta approval.

Auth & roles

  • Session = HMAC-signed cookie carrying the user id (lib/auth.ts); passwords scrypt-hashed with per-user salt.
  • requireUser(roles?) gates every page and action server-side; off-role users are redirected to their role home (teacher → /my-class, desk → /collect). Teachers can only save attendance for their own classRoomId.
  • Demo logins: loginAsDemo action + one-click buttons on /login. Remove for production.
Before any production deployment: real session store with revocation; MFA for staff (the PowerSchool breach began with one non-MFA credential); rate limiting on login; password reset; audit log on payments; DPDP Act consent records for guardian contact data; per-school row isolation (multi-tenancy).

Run it

  • Local: npm install → echo 'DATABASE_URL="file:./dev.db"' > .env → npx prisma db push → npm run seed → npm run dev. Logins: principal|teacher|desk@sunrise.school / demo123.
  • Vercel: switch prisma provider to postgresql, import repo, attach Neon Postgres (sets DATABASE_URL), set AUTH_SECRET, deploy, then db push + seed against the production URL. Full steps in README.
